# 10 分钟项目技术分享叙述稿

> 把 30 道面试题串成一条故事线，按项目开发的时间线展开。
> 面试官听完不会觉得你在"背题"，而是在"复盘一个真实的项目"。

---

## 开场（30 秒）

"我最近用 Vue 3 全家桶从零独立开发了一个仿小红书的移动端 H5 项目，叫 RedBook。代码在 GitHub 开源，部署在 GitHub Pages 可以直接体验。今天想按项目从搭建到上线的顺序，分享几个我印象最深的技术决策。"

---

## 第一幕：项目搭建 — 为什么选这套技术栈（1 分钟）

"拿到需求后第一步是技术选型。我选了 Vue 3 + Vite 7，而不是 Nuxt 或 React——因为这是移动端 H5，不需要 SSR，Vite 的构建速度和 HMR 体验是最好的。"

**引出问题 4（Composition API）**：

"而且 Vue 3 的 Composition API 特别适合这个项目。你看，项目里有点赞、收藏、关注三个功能，它们的逻辑几乎一模一样：先改 UI → 发请求 → 失败回滚。用 Options API 的话，同一功能的代码散落在 data、methods、watch 里，但用 Composable，我把这套逻辑抽成一个叫 `useToggleAction` 的函数——三个操作共享同一套代码，点赞只需要两行配置。"

**引出问题 3（竞态）**：

"这里面有个坑：用户可能会快速连点。所以我加了请求锁——`isPending` 标记，上锁后重复点击直接 return。"

**引出问题 8（Axios）**：

"说到请求，我选了 Axios 而不是 fetch。原因很实际——我需要请求拦截器自动带 Token、响应拦截器统一脱壳和错误处理、以及 timeout 超时配置。这些东西用 fetch 要自己封装，用 Axios 开箱即用。"

**引出问题 9（竞态）**：

"但 Axios 不能解决另一个问题：用户快速切换笔记页面时，旧页面的请求可能比新页面晚返回，导致数据覆盖。我的方案是递增 requestId——每个请求带一个版本号，响应回来时检查，版本号对不上就丢弃。配合 AbortController 直接取消旧请求更彻底。"

---

## 第二幕：移动端布局 — 瀑布流的 CSS 与性能权衡（2 分钟）

"接下来是 UI 部分。这个项目最核心的视觉特征是双列瀑布流。"

**引出问题 1（CSS Columns）**：

"我的第一版直接用了 CSS `columns-2`。好处是零 JS 代码，浏览器自动把卡片分配到两列。但它有个硬伤——排列顺序是'先从上到下填满第一列，再填第二列'，不是从左到右。而且你无法精确控制每一列的高度，网络不好时两列严重失衡。"

**引出问题 9（兼容性）**：

"好在兼容性很好——`column-count` 支持所有现代浏览器，全球覆盖率 97%。关键是要加 `break-inside: avoid`，防止单张卡片被切分到两列。"

**引出问题 2（IntersectionObserver）**：

"瀑布流还有第二个问题：可能有几十上百张图片。我写了 `v-lazy` 自定义指令，基于 IntersectionObserver 做懒加载——图片进入视口前 200px 开始加载，加载完后立刻 `unobserve` 释放资源。"

**引出问题 6（事件循环）**：

"Observer 回调的执行时机很有意思——它既不是宏任务也不是微任务，而是在浏览器每帧的 Layout 之后、Paint 之前执行。这意味着它不会阻塞渲染，但回调里的代码也不能太重。"

**引出问题 1（1000 张图）**：

"如果面试官问'1000 张图怎么办'——CSS Columns 就扛不住了，因为 1000 个 DOM 节点的内存和重排开销是 O(n) 的。这种场景必须上虚拟列表，只渲染视口附近约 20 张卡片，其他用占位 div 撑开高度。"

**引出问题 2（并发控制）**：

"还有个小细节——Observer 在快速滑动时会批量触发，可能同时发出十几二十个请求。我加了 loading 标记防止同一页重复请求，理想情况下还应该用 AbortController 取消用户已滑过区域的请求。"

**引出问题 6（骨架屏）**：

"图片加载期间呢，我用骨架屏做占位。关键点是骨架屏和真实卡片的尺寸必须一致——我给了每个图片容器 `min-height: 150px`，防止图片加载前卡片高度坍缩为 0。骨架屏消失和内容出现要无缝衔接，避免 Layout Shift。"

**引出问题 2（图片性能）**：

"更进一步的优化方向是 WebP 格式、CDN 按分辨率裁剪、渐进式 JPEG——但作为前端项目，懒加载 + 骨架屏已经解决了核心体验问题。"

## 第三幕：数据管理 — Mock 系统的设计哲学（2 分钟）

"这个项目没有后端，所有数据都是 Mock 的。但 Mock 怎么做，差别很大。"

**引出问题 4（Mock.js 原理）**：

"Mock.js 的方式是覆写浏览器原生的 `XMLHttpRequest`。这有个问题：Network 面板看不到请求，开发体验不好。而且如果页面上有其他库也覆写了 XHR，会冲突。"

**引出问题 4（字段兼容）**：

"所以我设计了两层 Mock：开发环境用 Vite 自定义中间件，在 Dev Server 层拦截请求，Network 面板可见；生产环境因为部署到 GitHub Pages 没有服务器，用 Axios 的 `config.adapter` 在浏览器端返回数据。"

"这里引申出一个问题：如果将来接真实后端，字段名变了怎么办？我的做法是在 API 层加 normalize 函数做 DTO 转换。后端改字段名，我只需要改 normalize 函数里的映射，所有组件无感。如果用 TypeScript 就更稳——interface 层做编译期检查。"

---

## 第四幕：状态同步与乐观更新（2 分钟）

**引出问题 5（Pinia 单例）**：

"状态管理我选了 Pinia。一个核心问题是：用户在详情页点了赞，回到首页后点赞状态要同步。Pinia 的 Store 实例是单例——内部通过一个 `Map<string, Store>` 缓存，不管你从哪个组件调 `useXxxStore()`，返回的都是同一个实例。所以详情页改状态，首页通过 `storeToRefs` 自动感知变化。"

**引出问题 6（请求锁超时）**：

"回到 `useToggleAction` 这个 Composable。除了请求锁，我还加了超时兜底。Axios 的 `timeout: 10000` 是主力防线，但我也准备了 `setTimeout` 的安全锁——8 秒后强制回滚并解锁，防止极端情况下的死锁。最后 `finally` 中的解锁是第三重保障，保证无论成功、失败还是超时，锁都能被释放。"

**引出问题 7（防抖 vs 节流）**：

"说到触发频率，Observer 和 scroll 事件不同——Observer 只在状态变化时触发（不可见→可见），不是持续触发的。所以我没用手动 debounce 或 throttle。Observer 的回调频率天然被绑在浏览器的帧率上。如果非要说 debounce 和 throttle 的区别——搜索框输入联想用 debounce（等用户停下来），滚动事件用 throttle（控制频率），Observer 本身已经是离散触发了。"

---

## 第五幕：存储、适配与部署（1.5 分钟）

**引出问题 10（localStorage 满）**：

"项目目前只有 Token 存 localStorage，量很小。但如果将来要缓存首页列表做离线浏览，localStorage 有 5MB 限制，满了会抛 `QuotaExceededError`。降级方案是 LRU 淘汰旧数据，或者冷数据搬去 IndexedDB。业界常用的 `pinia-plugin-persistedstate` 插件会根据数据量自动选择存储引擎。"

**引出问题 9（大屏适配）**：

"移动端适配方面，我没有用传统的 REM 方案，因为 REM 在大屏上元素会等比放大，体验很差。我的做法是用 Tailwind 响应式工具类 + `max-w-[1080px]` 限制内容最大宽度——手机上是全宽，平板上是居中窄条，更像 App。"

**引出问题 10（GitHub Actions）**：

"最后是 CI/CD。我配置了 GitHub Actions，push 到 main 分支自动触发——拉代码、装 Node、装依赖、Vite 构建、部署到 Pages。每次部署约 30 秒。如果失败，去 Actions 标签页展开日志就能定位。最常见的坑是 `package-lock.json` 没提交导致 `npm ci` 失败，或者 Pages 没在仓库设置里启用。"

**引出问题 9（Element Plus 体积）**：

"还有个优化点：Element Plus 全量引入约 600KB gzip，但项目其实只用了 `ElMessage`。如果要重构，按需引入就能减掉大部分体积，或者干脆手写一个 30 行的 Toast——不值得为一个提示框拖一个组件库。"

---

## 收尾（30 秒）

"这个项目的代码和五章学习教程都在 GitHub 上开源。在线演示地址也在 README 里。我觉得这个项目最让我有收获的不是某个具体 API，而是学会了'遇到问题→分析本质→选最合适方案'的思维方式。比如瀑布流，不是上来就用插件，而是先想 CSS 能不能做、局限性在哪、什么时候该升级到虚拟列表。这种'先理解问题、再选工具'的习惯，我觉得比具体技术更重要。"

## 面试临场技巧

**如果面试官打断你：**

- "这个点展开讲讲？" → 切到该题的详细回答，讲完再回主线
- "为什么不选方案 X？" → 先肯定方案 X 的适用场景，再说你场景下的取舍
- "你提到了 XX，具体怎么实现的？" → 切换到对应题目，结合项目代码讲解

**如果你卡住了：**

说"我可以打开项目代码演示一下吗？"——打开 GitHub，对照实际代码讲，比空口讲更有说服力，还能争取思考时间。
